## 0. Constraint restated

This cycle is **purely additive**, per Gino's explicit instruction, honored by all four scanners and by this consolidation. **No row in this review recommends cutting, simplifying, or removing anything Pondo already ships** — including any of the 27 shipped FRs or the Lent/Borrowed account-type model, even where the InnerGrow gap-table found that model is no longer a differentiator against InnerGrow specifically. That model remains valuable and still differentiates Pondo against YNAB/Monarch/Copilot per the prior BRD-era scan. Every row below evaluates only the **52 new ideas** surfaced this cycle — keep, absorb, kill, or needs-Gino's-input applies to the *new idea itself*, never to existing scope.

## 1. What the pool is really telling us

Stripped of the four scanners' different framings, this cycle converges on five real threads: **(1)** Pondo's dashboard is missing the single "headline number" (Net Worth) its own personas would actually open the app to check; **(2)** a cluster of cheap, high-trust fixes (privacy mask, balance reconciliation, pending flag, recycle bin) all attack the same BRD-identified #1 churn cause — drift between what the app says and what's real; **(3)** InnerGrow's mobile-native idioms (bottom nav, FAB, calendar view) translate cleanly to Pondo's responsive web with no native dependency, and the design scanner was careful to flag the handful that genuinely don't (D-15, D-16); **(4)** several already-BRD-deferred v2 items (recurring, budgets, CSV import, savings goals) got independently re-validated by a competitor doing exactly those things well — this cycle turns "deferred" into "spec'd, ready for a story"; and **(5)** the biggest-ticket ideas (household sharing, cloud sync, AI entry, reminders, native widget) all sit downstream of one decision Gino is already actively weighing — self-hosted vs. cloud — so this review keeps them as valid, Kept ideas but tags them clearly rather than pretending that decision is this cycle's to make.

**Triage result: 52 ideas in → 43 anchor ideas promoted to Keep user stories (folding in 6 more ideas absorbed into those same stories, so 49 of the 52 ideas are represented in the Keep section) → 8 recommended kill/absorb (2 kill, 6 absorb) → 1 idea + 1 framing-tension flagged for Gino's input. 52 accounted for: 43 + 6 + 2 + 1 = 52.**

**Two load-bearing findings that affect several rows below, found by direct inspection, not assumed from the SAD:**
1. **The SAD's stated data-layer decision (better-sqlite3) is not what's actually running.** `server/server.js` → `require('./db/schema')` → `server/db/schema.js` uses the **`sqlite3`** package (async, callback-style, `db.serialize()`), not `better-sqlite3`. This matters for every story below touching atomic writes (US-15, US-05) and for the cloud migration epic's data-layer sizing.
2. **`client/src/lib/api.js` hardcodes the app passphrase in source** (`const PASSPHRASE = 'pondo-app-secret'`), not stored in `sessionStorage` and prompted per the SAD §5.1 description. Today's "auth" is effectively a shared constant baked into the client bundle. **Also found, materially lowering three stories' effort:** `EmojiPicker.jsx` (icon-grid picker) and a native color-swatch input are **already fully wired** into `Categories.jsx` (create + edit forms, writing to the existing `icon`/`color` columns) and into `AddAccount.jsx` (emoji field, `accounts.emoji` column already in schema, already rendered in `AddEntry.jsx` and `FilterPanel.jsx` dropdowns). **US-10 and US-12 are effectively already shipped** — verify against AC, don't rebuild.

### 7.2 The cloud migration epic — sized as its own prerequisite line item

**This is what Gino explicitly asked for: a real plan to get Pondo from its current self-hosted single-user SQLite architecture to a cloud-hosted, multi-device one — not a tag on six stories.**

**What actually has to change (grounded in the real codebase, not the SAD's description of it):**

1. **Data layer.** The live app runs on the `sqlite3` package (async/callback), not `better-sqlite3` as the SAD states — confirmed by tracing the full require graph (§7 intro). `server/db/queries.js` is ~750 lines of raw, duplicated SQL with zero shared balance/query abstraction (confirmed by direct read — every aggregation function repeats its own `CASE`-logic inline). Whatever cloud target is chosen, this file gets substantially rewritten, not lightly adapted — the SAD's Tradeoff-1 sketch ("swap `better-sqlite3` for `pg`+`knex`") describes a migration path for a driver that isn't actually the one running today.
2. **Auth model.** Today's "auth" is a single passphrase hardcoded as a constant in `client/src/lib/api.js` (confirmed), checked against one `bcrypt` hash in a `settings` row server-side. Real multi-user auth — user table, session/JWT or magic-link, per-user identity, invite mechanism — is being built from zero, not extended.
3. **Data migration.** `pondo.db` (real, active) already has a documented history of silently diverging from a second file (`server/database.sqlite`, US-30) on this exact machine — direct evidence the schema-drift risk is not theoretical here. Any migration script needs a schema-version guard and a tested rollback, and US-30 should land first as cheap insurance.
4. **Deployment/ops.** `artifacts/06-runbook.md` is written entirely for Windows-desktop/localhost: start via `npm start`, stop via `Ctrl+C`, backup via manual `cp`, "monitoring" via reading the terminal (confirmed — `error-handler.js` only ever does `console.error`, no log aggregation exists). None of this transfers to a cloud target — secrets management, managed backups, real monitoring, and a deploy process all need to be built, not just re-pointed. **Useful head start found:** the runbook's own §11 already sketches two candidate targets — Cloudflare Pages+Workers+D1 (devops's own rough call: "Medium" effort, Express→Workers runtime change) and Cloudflare Pages+VPS (devops's own rough call: "Low" effort, SQLite stays as-is). These are early estimates from `devops`, not a committed decision — worth a short spike to pressure-test D1-compatibility specifically against the current `sqlite3`-callback code before committing.

**Effort band: XL. Risk: High.**

Risk drivers, all concrete (not generic "migrations are risky"):
- The SAD-vs-reality driver mismatch (finding #1 above) means the *documented* migration path assumes the wrong starting point; a fresh trace of the require graph (already done here) needs to inform whichever target is chosen.
- The already-observed divergent-database bug (US-30) shows this exact codebase has silently drifted once before — a real, not hypothetical, precedent for what goes wrong in an unguarded data migration.
- The hardcoded-passphrase-in-source auth model provides no real security boundary once reachable over the public internet — shipping to a cloud target before this is fixed is a genuine exposure, not just a UX gap.
